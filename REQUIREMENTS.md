# OpenPlotAgent· 需求说明书 v1

> 学术图表 AI agent 工作站 · 本地部署

---

## 一、产品定位

### 1.1 一句话定位

OpenPlotAgent 是一个**本地运行的学术图表 AI agent**，为论文写作者提供从实验数据收集到投稿级图表输出的完整工作流。

### 1.2 核心价值

不同于在线画图工具（Datawrapper、Flourish），OpenPlotAgent 聚焦于：

- **论文场景**：PGF 后端 + LaTeX 公式 + 投稿尺寸规范
- **agent 驱动**：Claude/本地 LLM 完成绝大部分代码生成与修改
- **可追溯**：所有代码、数据、对话都进 git，像 Overleaf 一样可回溯
- **可复用**：记忆系统让 agent 在同一项目/跨项目累积风格与偏好
- **端到端**：raw data 收集 → 清洗 → 出图 → PDF 导出，一站式

### 1.3 不做什么

- 不做云端协作（单机本地）
- 不做数据库和后台分析（只做图表）
- 不做真正的安全沙箱（信任 Claude 生成的代码，仅做轻度隔离）
- 不替代 VSCode/Jupyter（但提供"在 VSCode 打开"的出口）

## 二、核心概念模型

### 2.1 层级结构

```
User
 └─ Project  (≈ 一篇论文 / 一个研究课题)
     ├─ .venv/                       独立 Python 环境 (uv)
     ├─ .git/                        项目级 git 仓库
     ├─ PROJECT.md                   项目记忆
     ├─ _shared/                     跨 task 复用的样式、函数
     │   ├─ style.py
     │   ├─ palette.json
     │   └─ requirements.txt
     └─ tasks/
         └─ Task  (≈ 一个实验 / 一张图)
             ├─ TASK.md              任务记忆
             ├─ raw/                 原始数据 (Git LFS)
             ├─ processed/           画图用的表格 (git)
             ├─ chart/               plot.py + 输出
             ├─ conversations/       agent 对话记录
             └─ .OpenPlotAgent/          内部状态
                 ├─ context.json
                 ├─ archive/         压缩的历史对话
                 └─ stream.jsonl     实时数据接收缓冲
```

### 2.2 关键概念定义

| 概念 | 定义 |
|------|------|
| **Project** | 一篇论文或一项研究。对应一个 Python 环境、一个 git 仓库、一套共享样式。 |
| **Task** | Project 下的一个具体图表任务。对应一个实验、一组数据、一个 plot.py。 |
| **Raw data** | 实验收集的原始数据。可能是 jsonl / csv / parquet / 图像 / 日志。体积大，结构乱。 |
| **Processed data** | 从 raw 通过 pipeline 变换得到的画图用表格。结构规整，体积小，直接喂给 plot.py。 |
| **Pipeline** | `raw → processed` 的 Python 变换脚本。raw 更新时可重跑。 |
| **Version** | 一张图的某次迭代产出。一次 agent 生成 或 一次直接编辑 = 一个 version。对应一个 git commit。 |
| **Memory** | 三级 markdown 记录的偏好与上下文（Global / Project / Task）。 |
| **Agent turn** | agent 一次完整的工具调用 + 回复。可能包含多个工具调用。 |

## 三、功能需求

### 3.1 项目管理

**FR-PROJ-01 · 创建项目**
- 从空白创建：指定名称、可选描述、自动初始化 `.venv` 和 `.git`
- 从已有项目复制：可选继承样式（`_shared/`）、记忆（`PROJECT.md`）、图表作为模板、环境（`requirements.txt`）
- **不继承数据**（除非用户勾选）

**FR-PROJ-02 · 项目级环境**
- 每个 project 一个 `uv venv`，默认预装 `matplotlib, pandas, numpy, scipy, seaborn`
- agent 可以用 `install_package` 工具动态加包，自动写入 `requirements.txt` 并 git 提交
- 提供"环境重建"按钮：删除 `.venv` 重新 `uv pip install -r requirements.txt`

**FR-PROJ-03 · 项目级记忆 PROJECT.md**

```markdown
# Project: Nature Submission 2026

## 期刊与格式要求
- Nature single column: 88mm / 3.5 inch
- 字体：Times New Roman (LaTeX PGF)
- 最低 300 DPI

## 视觉约定
- 主色：#E63946 (实验组) / #457B9D (对照组)
- 所有图标题用 LaTeX 数学公式
- 图例默认右上，不带边框

## 通用偏好
- 每张图配一个 figure caption 的 tex 片段
- 出图后自动生成 .pdf 和 .png（预览用）

## 失败教训
- Cell 期刊紫粉配色不适合本主题，已弃用
```

**FR-PROJ-04 · 项目归档与导出**
- 打包整个项目为 zip（含 `.git`）
- 一键清理 `.venv` 后的瘦身打包
- 导出全部最终图为一个 PDF（可选封面页）

### 3.2 任务管理

**FR-TASK-01 · 创建任务**
- 名称（如 `Fig.2 时间演化`）
- 可选模板：从本项目其他 task 复制 plot.py 作为起点
- 自动创建目录结构和 TASK.md

**FR-TASK-02 · 任务级记忆 TASK.md**

```markdown
# Task: Fig.2 时间演化

## 数据背景
- 采集自 2026-03-15 的 12h 时序实验
- 每 30min 采样，共 24 个时点
- 关键指标：基因表达 X₁ (treatment) vs X₁_ctrl

## 本图重点
- 要突出 t=6h 的转折点
- y 轴对数刻度（数据跨 4 个数量级）
- 用户反复强调过要把 CI shaded band 画粗一点

## 用户决策轨迹
- [v2] 尝试过双 y 轴，用户否决
- [v5] 尝试过 violin plot，用户否决
- [v7] 最终用 line + shaded CI，用户批准

## 待办
- [ ] 补充 n=8 的第二批数据再重新出图
```

**FR-TASK-03 · 任务切换**
- 在左侧 Project Sidebar 切 task 时：
  - 保存当前 task 的 agent context
  - 加载目标 task 的 context + TASK.md
  - 切换到目标 task 的 sandbox（进入其 `.venv`）

### 3.3 数据管理

**FR-DATA-01 · Raw data 管理**

**支持格式**：`.csv, .tsv, .jsonl, .ndjson, .parquet, .json, .xlsx, .npy, .h5`

**来源**：
1. 用户上传文件（单个 / 批量 / 整个文件夹）
2. **API 实时接收**：每个 task 有唯一 endpoint，用户脚本打数据进来
3. 从其他 task 引用（软链接 + 记录来源）

**操作**：
- 预览（表格、图片、文本前 N 行）
- 元信息（尺寸、采集时间、schema）
- 打标签（用于后续筛选）
- Git LFS 跟踪，大文件也能追溯

**FR-DATA-02 · 实时数据接收（Ingest API）**

启动 task 时自动暴露端点：
```
POST http://localhost:8765/v1/ingest/{project_id}/{task_id}
Content-Type: application/x-ndjson / application/json

{"t": 0, "x": 1.23, "y": 4.56}
{"t": 1, "x": 1.45, "y": 4.78}
```

用户实验脚本示例：
```python
import requests
for step in experiment():
    requests.post(ENDPOINT, json={"t": step.t, "x": step.x})
```

UI 实时显示：接收计数、最近数据点预览、采样率。用户点 "快照" → 冻结当前 `stream.jsonl` 并 git commit。

**FR-DATA-03 · Pipeline: raw → processed**

用户/agent 写一个 `pipeline.py`：
```python
# pipeline.py – generated by agent
import pandas as pd, json

def run():
    raw = [json.loads(l) for l in open('raw/exp_001.jsonl')]
    df = pd.DataFrame(raw)
    df = df.groupby('t').agg(['mean', 'std']).reset_index()
    df.to_csv('processed/fig_data.csv', index=False)

if __name__ == '__main__':
    run()
```

- 用户也可通过**可视化方式**选数据生成 pipeline：选列、聚合、过滤，UI 把这些操作 translate 成 pandas 代码
- raw 更新时自动重跑 pipeline（可关）
- pipeline 是 git 追踪的一等公民

**FR-DATA-04 · Processed 表格的手动编辑**
- 与现有版本一致：可编辑单元格、插入行列、转置
- 编辑会创建 `processed/fig_data.csv` 的新 git commit
- 关键：手动编辑会**标记**该表格为 "manual override"，后续 pipeline 重跑会警告覆盖

### 3.4 Agent 核心

**FR-AGENT-01 · 可配置的 LLM 后端**

支持以下 provider：

| Provider | 类型 | 用途 |
|----------|------|------|
| Claude Code | local | 默认、最强 |
| Anthropic API | api | 无 Claude Code 环境时用 |
| Anthropic 兼容端点 | api-compat | 自部署 Claude 镜像 |
| Ollama | local | 本地 qwen-coder / llama 等 |
| OpenAI 兼容 | api-compat | DeepSeek、月之暗面、自部署 OpenAI API 等 |

**切换粒度**：
- 全局默认
- 项目级覆盖
- 单次 agent call 临时切换（用于对比）

**FR-AGENT-02 · 工具集（agent tools）**

以下工具供 agent 调用：

```
# 文件/目录
list_files(path)
read_file(path, start_line, end_line)
write_file(path, content)
str_replace(path, old, new)

# 数据
read_table(path) -> {schema, sample, shape, dtypes}
suggest_chart_type(table_schema) -> ["grouped_bar", "line", ...]

# 执行
execute_python(code, timeout=30) -> {stdout, stderr, artifacts}
install_package(name, version=None)
check_env() -> {python_version, installed_packages}

# Git
git_log(n=20)
git_diff(hash1, hash2, path=None)
git_restore(hash, path=None)
git_commit(message, auto=True)  # auto 模式由系统触发

# 记忆
memory_read(level: "global" | "project" | "task")
memory_write(level, section, content, mode="append" | "replace")

# 实时数据
ingest_status() -> {active, count, last_received_at, recent_points}
ingest_snapshot(label)  # 冻结快照 + git commit

# 图表渲染
render_chart() -> {svg, pdf_path, elements_map}  
# elements_map: {"title": {gid, bbox}, "legend": {...}, ...}

# 用户交互
ask_user(question, options=None)
```

**FR-AGENT-03 · Agent 执行流程**

每一轮 agent turn：

```
1. 构建 system prompt:
   - 角色说明
   - GLOBAL.md + PROJECT.md + TASK.md (按近度排序)
   - 当前 task 的文件 summary
   - 可用工具列表
   - 最近 N 轮对话 (压缩过的旧对话作为一条 summary)

2. 发送到选定的 LLM provider

3. 接收流式回复，识别工具调用

4. 执行工具调用 (在 project 的 .venv 沙箱内)
   - 结果喂回 agent
   - 可能多轮工具调用

5. agent 给出最终回复 + 产出 (plot.py / pipeline.py / 图片)

6. 自动 git commit (防抖 1.2s)

7. 更新 UI: 预览、版本时间轴、聊天记录
```

**FR-AGENT-04 · 上下文压缩**

触发条件：
- 对话累积 token > 阈值（用户可配，默认 60k）
- 或：task 切换导致的状态保存

压缩策略：
```
1. 保留最近 20 轮完整对话
2. 之前的对话用 LLM 生成结构化摘要:
   - 用户核心诉求列表
   - 已尝试的方案和结果
   - 最终决策和理由
3. 摘要追加到 TASK.md 的 "对话摘要" 章节
4. 原对话归档到 .OpenPlotAgent/archive/<timestamp>.jsonl
5. 新的 agent prompt 使用 摘要 + 最近 20 轮
```

UI 显示"上下文使用率"进度条，接近阈值时提示用户"即将压缩旧对话"。

**FR-AGENT-05 · 错误恢复**

agent 执行 `execute_python` 失败时：
1. 把 stderr 喂回 agent，要求自我修复（最多重试 2 次）
2. 仍失败则交给用户：显示错误 + 最后一次尝试的代码 + 建议
3. 用户可选"手动修改代码"、"重新描述需求"、"回滚"

### 3.5 图表编辑

**FR-CHART-01 · Agent 驱动的生成**
（对话框输入自然语言需求 → agent 生成 plot.py）

**FR-CHART-02 · 直接操作**

预览区 SVG 每个元素带 gid，支持以下交互：

| 元素 | 交互 | 映射到 plot.py 的修改 |
|------|------|----------------------|
| 标题 / 轴标签 / 图例文字 | 双击 → LaTeX 编辑器 | 修改对应字符串字面量 |
| 图例 | 拖动 | `bbox_to_anchor` 参数 |
| 任意柱子 / 线 / 点 | 点击 → 属性小面板 | `color / alpha / hatch / linewidth / marker` |
| 坐标轴 | 双击刻度 → 改范围 | `set_xlim / set_ylim` |
| 整图尺寸 | 拖角 | `figsize` |
| 文字字号 | 元素右键 → 菜单 | `fontsize` |

**实现**：agent 不负责微修改，由前端直接 patch plot.py（通过 AST 修改或简单字符串替换），后端重渲染。每次直接操作触发 1.2s 防抖后的 git commit。

**FR-CHART-03 · 代码视图**

右侧面板 "Python 代码" 标签页：
- 只读模式：语法高亮 + 行号
- 编辑模式：可修改代码，"保存并重新渲染"触发重跑
- "在 VSCode 打开"：`code <path>` 调出 VSCode

**FR-CHART-04 · 版本管理**

每个 task 内部：
- 每一轮 agent 生成 = 一个 version
- 每一次直接操作（防抖后）= 一个 version
- 版本时间轴显示缩略图 + 操作描述 + 时间
- 支持对比（并排）、恢复、打星、删除、设为最终版

**FR-CHART-05 · 最终版与导出**

"设为最终版"：
- Git 打标签 `final-<task_id>-v<N>`
- 导出 PDF（PGF 后端渲染 LaTeX）
- 可选：提取样式到 `_shared/`、归档其他 version

### 3.6 Git 版本管理

**FR-GIT-01 · 自动提交**

触发动作 → commit message 模板：

| 动作 | Message |
|------|---------|
| 上传 raw 数据 | `raw: add 3 files (exp_001..exp_003)` |
| API 实时接收快照 | `raw: snapshot "n=100, t=12h"` |
| Pipeline 运行 | `processed: run pipeline (2 outputs)` |
| 手动编辑表格 | `processed: edit 5 cells` |
| Agent 生成代码 | `plot: v3 – <user request summary>` |
| 直接操作图表 | `plot: move legend (-24,+8)` |
| 换配色 | `style: apply Okabe-Ito` |
| 记忆更新 | `memory: update PROJECT.md §palette` |
| 设为最终版 | `finalize: v4 [TAG: final-fig2-v4]` |

所有 commit 防抖 1.2s 合并。

**FR-GIT-02 · History 面板**
- Overleaf 风格的时间线
- 每条 commit 带类型徽章（FINAL / AI / EDIT / STYLE / DATA）
- 点击 commit：展开 → 恢复 / 对比 / diff
- 对比模式：左右并排显示两个 commit 的图
- Diff：显示 plot.py 的文本 diff 和 processed/fig_data.csv 的表格 diff

**FR-GIT-03 · 分支**
- 默认不显式暴露分支概念
- "试试别的方案"按钮 = 创建隐式分支 `try-<timestamp>`
- 用户满意后 "合并回 main" 或 "丢弃"

### 3.7 记忆系统

**FR-MEM-01 · 三级 Markdown**

```
~/OpenPlotAgent/GLOBAL.md                        # 跨项目偏好
~/OpenPlotAgent/projects/<project>/PROJECT.md    # 项目偏好
~/OpenPlotAgent/projects/<project>/tasks/<task>/TASK.md  # 任务偏好
```

**FR-MEM-02 · 自动更新**

agent 用 `memory_write` 工具更新。触发场景：
- 用户明确表达偏好："以后都用 Okabe-Ito 配色" → 写入 PROJECT.md
- 用户否决了某个方案："不要双 y 轴" → 写入 TASK.md
- 成功定稿时：总结决策过程写入 TASK.md

**FR-MEM-03 · 手动编辑**

UI 提供 Memory 面板（tabs 中的一个），显示三份 md 的内容，用户可直接编辑。编辑触发 git commit。

**FR-MEM-04 · 记忆可视化**

每次 agent turn 时，UI 顶部短暂显示"本次加载的记忆"：
```
使用记忆：GLOBAL (47L) · PROJECT (23L) · TASK (15L)
```
点击展开看具体注入了什么。

## 四、非功能需求

### 4.1 性能

- 预览区图表重绘 < 500ms（不含 LLM 生成时间）
- 表格粘贴解析 < 200ms（1 万行以下）
- 项目切换 < 1s（含 venv 激活）
- LLM 响应首 token < 2s（本地模型）/ 3s（API）

### 4.2 安全性

- LLM 生成的代码在 project 的 `.venv` 内执行
- `execute_python` 强制 30s 超时
- 禁止网络访问（除了 `install_package` 专门通道）
- 禁止访问 `~/OpenPlotAgent/` 以外的路径（通过 cwd 限制）
- API key 加密存储（macOS Keychain / Linux Secret Service）

### 4.3 可靠性

- 所有编辑操作先写临时文件，成功后原子替换
- Agent 执行中断恢复：保存中间状态到 `.OpenPlotAgent/context.json`
- Git 操作失败不影响用户（后台重试）

### 4.4 可扩展性

- 工具系统可插件化：新增 tool 只需实现 `@tool` 装饰器的函数
- LLM provider 可插件化：实现 `LLMProvider` 接口即可接入新模型
- 导出格式可插件化：PDF / PNG / SVG / TikZ 可加新后端

## 五、技术栈

### 5.1 前端

- **框架**：React 18 + Vite (HMR 热重载)
- **样式**：Tailwind CSS + shadcn/ui
- **状态**：Zustand (简单) 或 Jotai (原子化)
- **LaTeX 渲染**：KaTeX (交互预览) / 后端 matplotlib PGF (终稿)
- **代码编辑**：Monaco Editor
- **图表 SVG 操作**：d3-drag + 自写 gid 匹配

### 5.2 后端

- **框架**：FastAPI + uvicorn --reload
- **Agent 循环**：自写或 Anthropic SDK 的 tool_use 格式
- **沙箱**：`uv` + `subprocess` + `resource` 限制
- **Git**：GitPython + Git LFS (raw data)
- **实时数据**：FastAPI WebSocket + Server-Sent Events
- **持久化**：纯文件系统（md + json + csv），无数据库

### 5.3 LLM 抽象

```python
class LLMProvider(ABC):
    @abstractmethod
    async def chat(self, messages, tools) -> AgentResponse: ...

class ClaudeCodeProvider(LLMProvider): ...
class AnthropicAPIProvider(LLMProvider): ...
class OllamaProvider(LLMProvider): ...
class OpenAICompatProvider(LLMProvider): ...
```

### 5.4 部署

- 本地跑：`OpenPlotAgent start` 启动 FastAPI + 自动开浏览器到 `localhost:3000`
- 配置文件：`~/OpenPlotAgent/config.toml`
- 数据目录：`~/OpenPlotAgent/projects/`
- 一键安装：`pip install OpenPlotAgent` 或 `curl ... | bash`

## 七、附录：工作流典型场景

### 场景 A · 从零出一张论文图

1. 新建 project "Nature 2026"，选择单栏模板（继承字体、尺寸、DPI 默认值）
